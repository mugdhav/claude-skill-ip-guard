# ip-guard

**Behavioral guardrails for IP-safe code generation, content creation, and artifact building with Claude.**

---

## What Is This?

`ip-guard` is a Claude skill that adds a three-stage intellectual property compliance layer to every generation task. It activates automatically whenever Claude is about to write code, suggest dependencies, generate artifacts, or produce content for commercial use.

Think of it as a senior engineer and a cautious lawyer looking over Claude's shoulder at every step — one checks if the code is clean, the other checks if it's legally safe.

> **Analogy:** Existing license audit tools (like `license-checker` or Snyk) are smoke detectors — they alert you *after* potentially infringing code is already in your codebase. `ip-guard` is a building code inspector present *during* construction.

---

## Why This Skill Exists

Vibe-coded apps are starting to look the same. Same layouts, same interactions, same feel. That's the visible symptom everyone notices. The invisible risk is what's underneath.

When AI builds your app, it makes hundreds of small decisions you never see: which libraries to pull in, which patterns to implement, which assets to use. Most of the time, nobody asks what license any of it ships under. AI coding assistants don't apply license compliance reasoning by default — and unlike a human developer who intuitively knows not to copy-paste a GPL library into a proprietary codebase, they won't flag it unless explicitly asked.

These real failure modes drove this skill's design:

1. **The silent dependency problem.** Claude adds a library to your commercial project without declaring its license or checking compatibility. You ship. Months later, legal finds a GPL dependency in your proprietary codebase — a license incompatibility that should have been caught before a line of code was written. ip-guard addresses this by requiring Claude to declare a dependency plan with compatibility status and wait for your approval before proceeding.

2. **The content provenance gap.** Generated documentation, UI copy, and marketing content carries no audit trail by default. If content was reproduced verbatim from a copyrighted source during generation, there is no record and no flag. ip-guard instructs Claude to paraphrase rather than reproduce, and appends a provenance block to every artifact recording what sources were referenced and what needs human review.

`ip-guard` addresses both — keeping human oversight at every decision point where IP risk enters a commercial build.

## What the Skill Checks For

### Stage 1 — Pre-Generation (Before Claude writes anything)

| Check | What it does |
|---|---|
| License target declaration | Reads existing `package.json`, `LICENSE`, or `CLAUDE.md` to infer project license, or asks the user. Never creates or modifies a LICENSE file autonomously. |
| Dependency intent scan | Lists every library Claude plans to use, with license and compatibility status, before writing code |
| Asset source check | Flags fonts, icons, and images from unverified or commercial-only sources |

### Stage 2 — Inline Flagging (During generation)

| Signal | Claude's action |
|---|---|
| Open Source dependency with a restrictive license (GPL, AGPL) incompatible with project license target | Flags incompatibility and pauses before adding |
| New import not in the pre-gen plan | Pauses, declares it with license and compatibility before proceeding |
| Reproducing >10 lines from identifiable source | Refuses; rewrites generically; cites origin |
| Algorithm associated with a known patent | Adds `⚠️` comment recommending pre-commercial legal review |
| Verbatim content from fetched sources | Paraphrases with attribution; never reproduces |

### Stage 3 — Post-Generation (After every artifact)

Every artifact — code file, document, UI component, or content piece — gets an appended **IP Provenance Block**:

```
---
## 🛡️ IP Provenance Summary
Generated by: ip-guard skill v1.0
Date: 2026-03-17
Mode: Production

Project license target: MIT

Dependencies added this session:
| Package   | Version | License     | Compatible |
|-----------|---------|-------------|------------|
| axios     | 1.6.0   | MIT         | ✅         |
| sharp     | 0.33.0  | Apache-2.0  | ✅         |

Patterns used:
- Standard REST client pattern — generic algorithmic knowledge
- Image resizing via sharp API — documented usage pattern

Assets used: None

⚠️ ITEMS REQUIRING HUMAN REVIEW: None

Disclaimer: This summary supports audit trails but is not legal advice.
---
```

## Modes

### Production Mode (default)
Full three-stage workflow. Pre-gen checks run, inline annotations apply, full provenance block appended. Best for client work, commercial products, open source releases.

### Fast Mode
Triggered by: "fast mode", "prototype", "quick draft", "just experiment"

Pre-gen checks and inline annotations are skipped. Abbreviated provenance block still appended. Best for throwaway experiments and personal learning projects.

```
---
## 🛡️ IP Provenance (⚡ Fast Mode)
Date: 2026-03-17 | Target license: unknown
Dependencies: axios (MIT), lodash (MIT)
⚠️ Review: None — fast mode, pre-gen checks skipped
---
```

## What the Skill Doesn't Do

Being clear about limitations is important:

| Limitation | Why |
|---|---|
| **Cannot inspect Claude's training data sources** | Claude cannot introspect what it learned from |
| **Cannot do live patent database lookups** | USPTO/EPO APIs are not connected — flags *patterns* only |
| **Cannot guarantee generated code is not similar to training data** | Provenance of generated code cannot be fully verified |
| **Not a substitute for legal counsel** | All provenance blocks include this disclaimer |

For deeper vetting, pair this skill with an n8n workflow calling FOSSA, Snyk, or Copyleaks APIs.

## Cost vs. Benefit

The overhead is real and worth stating clearly. So is the return.

### The cost

**Additional workflow steps:**

```
Without ip-guard:   Request → Claude generates → Done (3 steps)

With ip-guard:      Request → Pre-gen check → [Your confirmation] →
                    Claude generates (with inline flags) →
                    Post-gen provenance block → [Review ⚠️ items] → Done
                    (5–7 steps; 2 require your input, only when needed)
```

**Token overhead per session:**

| Component | Approx. tokens |
|---|---|
| Skill instructions (loaded per session) | ~800–1,200 (input) |
| Pre-gen checklist output | ~200–400 per generation request |
| Inline license annotations | ~100–300 (scattered through code) |
| Provenance summary block | ~150–250 (output) |
| **Total per session** | **~1,250–2,150 (~5–10% of a typical session)** |

**Time overhead:**

| Step | Added time |
|---|---|
| First-time license declaration (per project) | 30–60 seconds (one-off) |
| Pre-gen dependency scan | 10–20 seconds per major generation |
| Provenance block appended | 5–10 seconds (automatic) |
| Reviewing ⚠️ items | 0 seconds if none; 2–5 min if triggered |

The overhead is **front-loaded** — heaviest at project setup, nearly invisible in day-to-day generation once the license target is established.

---

### The benefit

The protection is **permanent and compounding**.

**Catches problems before they're expensive.** A GPL dependency discovered during legal review before a commercial release costs an afternoon to fix. The same discovery six months post-launch — after the product is shipped, the codebase is deep, and a client is asking questions — costs significantly more in time, legal fees, and credibility.

**Creates an audit trail that strengthens your copyright claim.** Under current law in most jurisdictions, AI-generated code may not qualify for copyright protection unless there is meaningful human authorship and direction. The provenance block documents exactly that: you directed the work, you reviewed the output, you made editorial decisions. That record matters.

**Normalises IP hygiene in AI-assisted workflows.** The biggest risk in AI-assisted development isn't one bad decision — it's the accumulation of hundreds of small unreviewed decisions. ip-guard makes compliance a default, not a special project.

**Scales with your risk profile.** Low-stakes prototype? Use fast mode — minimal overhead, still leaves an audit trail. Commercial client work? Full production mode. The cost scales down; the protection scales up.

> Think of it like writing tests. Tests slow the current sprint slightly but prevent a multi-day debugging session six weeks later. The ROI on ip-guard is highest exactly when the stakes are highest: client work, commercial products, open source releases.

## Installation

### Claude Code / Claude Desktop

1. Download and unzip the skill
2. Copy the `ip-guard/` folder to:
   - **Personal (all projects):** `~/.claude/skills/ip-guard/`
   - **Project-specific:** `.claude/skills/ip-guard/` (in your project root)
3. Claude will automatically detect and load the skill

### Claude.ai (Web/Mobile)

1. Go to **Settings → Capabilities → Skills**
2. Upload the `ip-guard.skill` package

## Running a Full License Audit

For a post-session audit of all project dependencies:

```bash
# From your project root:
bash .claude/skills/ip-guard/scripts/license_audit.sh \
  --project-license MIT \
  --output license-report.md
```

Supports: **Node.js** (via `license-checker`), **Python** (via `pip-licenses`), **Rust** (via `cargo-license`), **Go** (via `go-licenses`).

## What's Not in This Skill (But Could Be)

This skill is intentionally scoped to behavioral guardrails. A companion automation layer could extend it with:

- Live FOSSA/Snyk API calls for real-time vulnerability + license data
- USPTO patent search via n8n workflow
- Copyleaks API integration for content originality scoring
- Automatic SBOM (Software Bill of Materials) generation on every commit

Contributions welcome.

---

## Known Limitations & Caveats

- The compatibility matrix covers the most common licenses. Edge cases (dual-licensed packages, custom license agreements, CC license variants) require manual review.
- Patent flagging is pattern-based, not database-backed. It flags *categories* of algorithms known to have patent history (compression, image processing, cryptography) — not specific patents.
- This skill is not legal advice and explicitly says so in every provenance block.
